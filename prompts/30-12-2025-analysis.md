✦ Your game engine, g10, is a sophisticated, data-driven system built on top of SDL3 and a custom foundation library (GSDK). It strikes a balance between
  low-level performance (C) and high-level flexibility (JSON/Python).

  Architectural Analysis

  1. Data-Driven Core
  The engine is almost entirely configured via JSON. instance.json acts as the manifest, defining everything from window parameters and input bindings to the
  complex structure of the rendering pipeline.
   * Initialization: g_init orchestrates the startup, handing off platform-specific work to the WSI (Window System Integration) layer.
   * Dynamic Renderer: The renderer is not hardcoded. It constructs attachments (textures/framebuffers), pipelines (shaders/state), and passes based on the
     JSON description. This allows you to change the rendering technique (e.g., switching from Forward to Deferred) primarily by editing assets.

  2. Rendering Pipeline (SDL3 GPU)
  You are utilizing the brand-new SDL3 GPU API, which provides a modern, Vulkan/Metal-like abstraction.
   * Pipeline-Centric Drawing: Instead of a traditional scene-graph traversal during the draw call, entities register themselves with specific pipelines
     during loading (entity_from_json).
   * Batching & State Management: In g_sdl3_render_pass_draw, the engine iterates through pipelines. By binding a pipeline once and then drawing all
     associated entities, you naturally minimize expensive GPU state changes.
   * Uniform Management: The uniform_set_pack_push system is efficient. It uses a "pack" function pointer approach to serialize C structs (like mat4)
     directly into a buffer that is then pushed to the GPU command buffer.

  3. Asset Pipeline & Blender Integration (gport)
  The integration with Blender is a highlight of the project.
   * Headless Export: Using a Makefile to trigger headless Blender processes for asset export is a professional-grade workflow. It ensures that the "source
     of truth" (the .blend file) is always in sync with the engine's optimized JSON formats.
   * Custom Geometric Data: Your Geometry class in Python handles the heavy lifting of calculating normals, tangents, and bitangent signs, which are
     essential for advanced shaders like the "tbn" (Normal Mapping) pipeline.
   * Scene Reflection: The Scene class mirrors the Blender hierarchy into the engine, allowing you to design levels visually and see them reflected in
     scene.json.

  4. Game Logic & Control
   * Input Mapping: The input system handles rebindable keys and maps them to named actions (e.g., "FORWARD"). This decouples game logic from specific
     hardware scancodes.
   * Callback Architecture: The program_pipeline and user_code_set_callback pattern allows the game layer (lightspeed) to "plug in" logic at specific points
     in the engine's lifecycle without modifying the core engine code.

  Observations & Potential Improvements

   * Shader Portability: The compile-metal-shader.sh script is currently hardcoded for macOS/Metal. To achieve the cross-platform promise of SDL3, you might
     eventually want to look into SDL_shadercross or a similar tool to compile SPIR-V/HLSL/MSL from a single source.
   * Memory Management: The engine makes extensive use of a default_allocator. Ensuring these are tracked (perhaps via a memory profiler in GSDK) will be
     important as the scene complexity grows.
   * Resource Caching: You have a solid cache system (p_instance->cache). Currently, entity_from_json seems to load textures and geometry. Ensuring that
     multiple entities sharing the same texture don't reload it from disk would be a good optimization if not already fully handled by GSDK's cache.
   * The "Static" Draw List: Entities are added to a p_static_draw_list. If you plan on having many dynamic entities (bullets, particles) that are
     created/destroyed frequently, you may need a "dynamic" draw list path to avoid the overhead of constant array manipulation in the static list.

  Overall, the architecture is very clean and follows modern "Render Graph" inspired principles while remaining accessible. Would you like to dive deeper
  into a specific system, such as the SDL3 GPU implementation or the GSDK data structures?
